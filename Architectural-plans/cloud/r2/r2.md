# R2

Archivo: `Architectural-plans/cloud/r2/r2.md`

## Propósito

`R2` define el almacenamiento de objetos de Lumen.

R2 guarda archivos grandes o binarios que no deben vivir en D1: paquetes de ejercicios, assets visuales, imágenes, materiales, bundles y recursos descargables.

La regla principal es:

```txt
D1 guarda metadata.
R2 guarda objetos.
Workers API controla acceso.
```

## Qué guarda R2

R2 debe guardar:

- paquetes de ejercicios
- bundles de ruta
- assets de nodos
- imágenes decorativas
- materiales WebGL
- presets exportados
- archivos grandes de módulos
- recursos versionados
- posibles archivos `.esex`

R2 no debe guardar progreso local del usuario.

## Qué no guarda R2

R2 no debe guardar:

- progreso local
- intentos locales
- errores locales
- estado de ruta del usuario como fuente principal
- datos temporales de UI
- metadata relacional compleja

Eso pertenece a SQLite local o D1, según sea local o cloud.

## Tech stack del submódulo

- **Cloudflare R2**: object storage.
- **R2 buckets privados**: almacenamiento no público por default.
- **Workers R2 binding**: acceso desde Workers API.
- **Object keys versionados**: rutas de objetos estables y versionadas.
- **Bucket scoped tokens**: control granular si se necesita administración externa.
- **S3-compatible APIs**: posible uso futuro con tooling compatible.

## Buckets iniciales

Estructura conceptual:

```txt
lumen-packages
lumen-assets
```

También puede ser un solo bucket con prefijos claros:

```txt
packages/
assets/
routes/
modules/
materials/
```

La decisión final puede ajustarse, pero los objetos deben estar ordenados y versionados.

## Paquetes de ejercicios

Los paquetes de ejercicios deben vivir en R2.

Ejemplo:

```txt
packages/c/strings/hello-world/v1/hello-world.esex
```

Un paquete puede incluir:

- enunciado
- archivos iniciales
- metadata
- tests si existen
- assets específicos
- restricciones
- conceptos
- versión

D1 guarda la metadata y la key del paquete.

R2 guarda el paquete.

## Assets visuales

R2 debe guardar assets visuales que la UI necesita descargar o cachear.

Ejemplos:

```txt
assets/routes/c/strings/node-active/v1.png
assets/routes/c/strings/node-locked/v1.png
assets/routes/c/strings/decor-abc/v1.png
materials/snake/green/body.png
materials/snake/green/cap-left.png
materials/snake/green/cap-right.png
```

Esto no significa que la UI dependa solo de PNGs.

La UI puede renderizar snake path con WebGL y generar sombras matemáticamente.

R2 solo guarda assets cuando conviene versionarlos o descargarlos.

## Privacidad

Los buckets R2 deben ser privados por default.

Los paquetes de ejercicios no deben servirse como archivos públicos sin control.

Workers API debe decidir si un paquete o asset puede entregarse.

Esto permite proteger ejercicios bloqueados y evitar que Free Mode descargue ejercicios de ruta antes de tiempo.

## Descarga autorizada

El cliente no debe descargar paquetes privados directamente sin pasar por Workers API.

Flujo esperado:

```txt
Local Engine pide paquete.
Workers API valida metadata y permiso.
Workers API lee R2 o entrega acceso autorizado.
Local Engine descarga/materializa localmente.
```

El estado local de desbloqueo se valida antes de materializar.

## Versionado de objetos

Todo objeto importante debe tener versión.

Esto evita sobrescribir contenido de forma peligrosa.

Ejemplo:

```txt
hello-world/v1/hello-world.esex
hello-world/v2/hello-world.esex
```

Si cambia un paquete, Lumen debe poder distinguir versiones.

No debe reemplazar trabajo local del usuario sin política explícita.

## Relación con D1

D1 guarda metadata y referencias.

R2 guarda los objetos.

Ejemplo:

```txt
D1:
  exerciseId: strings-basic
  packageKey: packages/c/strings/strings-basic/v1/package.esex

R2:
  packages/c/strings/strings-basic/v1/package.esex
```

## Relación con KV

KV puede cachear manifests que apuntan a objetos R2.

KV no debe guardar el objeto grande.

## Relación con Exercise Collection

La colección usa metadata de D1 para saber qué ejercicios existen.

Cuando el usuario importa un ejercicio disponible, el Local Engine pide el paquete R2 mediante Workers API.

Si el ejercicio está bloqueado, no debe descargar.

## Relación con UI de rutas

R2 puede guardar assets usados por la UI de rutas.

Pero la UI de rutas debe renderizarse localmente.

R2 no es el renderer.

R2 solo almacena recursos descargables y versionados.

## Fallos esperados

R2 debe contemplar:

- objeto no encontrado
- bucket no disponible
- permiso denegado
- descarga incompleta
- paquete corrupto
- versión inexistente
- asset faltante
- problema de CORS si algo se hace público
- conflicto de versión

Todos deben llegar a Lumen como errores controlados.

## Reglas deterministas

R2 guarda objetos.

D1 guarda metadata.

Workers API controla acceso.

R2 es privado por default.

Los paquetes de ruta bloqueados no se descargan antes de desbloquearse.

Los objetos importantes son versionados.

R2 no guarda progreso local.

## Resultado esperado

R2 permite que Lumen tenga ejercicios, assets y bundles remotos sin usar GitHub Releases como registry final.

El usuario descarga solo lo que necesita.

Lo importado se materializa localmente.

La UI puede usar assets versionados.

Las rutas pueden crecer sin meter todos los recursos dentro de la extensión.
