# D1

Archivo: `Architectural-plans/cloud/d1/d1.md`

## Propósito

`D1` define la base de metadata cloud de Lumen.

## Estado actual del repo

D1 no está implementado en este repositorio. No hay migraciones cloud, schema,
bindings ni Worker que consulte D1. Este documento describe la arquitectura
objetivo.

D1 no es la base local del usuario. No reemplaza SQLite local. No debe ser la fuente principal del progreso local.

D1 guarda metadata estructurada remota: colección de ejercicios, rutas, módulos, versiones, paquetes, assets y reglas generales de contenido.

La regla principal es:

```txt
D1 guarda metadata cloud.
SQLite local guarda estado del usuario.
```

## Qué guarda D1

D1 debe guardar datos estructurados que Cloud necesita consultar.

Ejemplos:

- rutas disponibles
- módulos de cada ruta
- ejercicios de la colección
- relación ejercicio-ruta
- relación ejercicio-módulo
- estado base de bloqueo
- versiones de ejercicios
- versiones de rutas
- manifests
- referencias a paquetes R2
- referencias a assets R2
- metadata visual básica
- reglas de desbloqueo generales

## Qué no guarda D1

D1 no debe guardar como fuente principal:

- código del usuario
- intentos locales completos
- errores locales completos
- progreso local obligatorio
- archivos `.c`
- builds
- estado visual local
- conversaciones completas de la Guía por default

Si en el futuro existe sincronización cloud, debe ser un módulo aparte y no cambiar esta regla base.

## Tech stack del submódulo

- **Cloudflare D1**: base cloud serverless con semántica SQL de SQLite.
- **SQL migrations**: control de evolución del schema cloud.
- **Workers binding de D1**: acceso desde Workers API.
- **Wrangler D1 commands**: creación, migración y operaciones de desarrollo.
- **SQLite-compatible schema mindset**: mantener conceptos compatibles con SQLite local cuando tenga sentido.

## Entidades principales

Entidades conceptuales:

```txt
routes
modules
exercises
exercise_versions
route_modules
module_nodes
exercise_packages
assets
manifests
unlock_rules
visual_presets
```

No hace falta crear una documentación separada para cada entidad todavía.

La implementación puede empezar con lo mínimo necesario para Ruta C y crecer.

## Rutas

D1 debe poder representar rutas.

Ejemplo conceptual:

```txt
routeId: c
title: Ruta C
description: Fundamentos de programación en C
version: 1
```

La ruta no contiene el progreso del usuario.

Contiene la definición remota de la experiencia.

## Módulos

D1 debe representar módulos de ruta.

Ejemplo:

```txt
moduleId: strings
routeId: c
title: Cadenas de caracteres
subtitle: char, strings y texto
order: 2
themeColor: cyan-green
```

La metadata visual puede vivir parcialmente aquí, pero los assets grandes viven en R2.

## Nodos de ruta

D1 puede guardar la definición de nodos de un módulo.

Cada nodo puede apuntar a un ejercicio, reto, quiz o proyecto.

Ejemplo conceptual:

```txt
nodeId
moduleId
exerciseId
nodeType
order
pathPosition
stateRule
```

D1 define la estructura base.

El estado real de completado, activo o bloqueado se combina con SQLite local.

## Ejercicios

D1 debe guardar metadata de ejercicios.

Ejemplo:

```txt
exerciseId
title
description
language
difficulty
concepts
source
routeLocked
packageId
version
```

El enunciado completo puede vivir en el paquete de R2 si conviene.

D1 guarda lo suficiente para listar, filtrar y decidir qué paquete descargar.

## Bloqueos

D1 puede definir que un ejercicio pertenece a una ruta y que requiere desbloqueo.

Pero D1 no decide si el usuario local ya lo desbloqueó.

La regla es:

```txt
D1 define reglas.
Local Engine aplica reglas.
SQLite local guarda estado del usuario.
```

## Metadata visual

D1 puede guardar metadata visual pequeña:

- color del módulo
- preset visual
- referencia a snake path preset
- número de nodos
- referencias a assets
- posiciones o parámetros de nodos si son datos ligeros

D1 no debe guardar imágenes grandes ni binarios.

Eso pertenece a R2.

## Manifests

D1 debe poder construir o alimentar manifests.

Un manifest resume qué contenido existe y qué versiones están disponibles.

Ejemplos:

```txt
collection-manifest
route-c-manifest
module-strings-manifest
assets-manifest
```

Los manifests pueden cachearse en KV si se leen mucho.

## Relación con Workers API

Workers API consulta D1.

El cliente no debería hablar directamente con D1.

Workers API valida, transforma y devuelve respuestas estables.

## Relación con R2

D1 guarda referencias a objetos en R2.

Ejemplo:

```txt
packageKey: packages/c/strings/hello-world/v1.esex
assetKey: assets/routes/c/strings/node-active/v1.png
```

R2 guarda el objeto.

D1 guarda la referencia y metadata.

## Relación con KV

KV puede cachear consultas frecuentes derivadas de D1.

D1 sigue siendo la fuente de verdad.

KV solo acelera lectura.

## Reglas deterministas

D1 es metadata cloud.

D1 no reemplaza SQLite local.

D1 no guarda paquetes ni assets grandes.

D1 no decide progreso local por sí solo.

D1 define rutas, módulos, ejercicios, manifests y reglas generales.

D1 debe usar migraciones.

D1 debe estar detrás de Workers API.

## Resultado esperado

D1 permite que Lumen tenga una colección remota organizada.

Permite versionar Ruta C.

Permite saber qué módulos y ejercicios existen.

Permite referenciar paquetes y assets en R2.

Permite que Workers API responda con datos estructurados sin hardcodear todo en código.
